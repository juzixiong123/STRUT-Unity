## 三、失败原因分析

| 失败原因 | 数量 | 占比 |
|----------|------|------|
| **空 stderr（未知错误）** | 48 | 42.9% |
| **Oracle 编译错误** | 35 | 31.3% |
| **其他错误** | 29 | 25.9% |
| **返回类型不支持** | 5 | 4.5% |
| **Oracle 运行时崩溃** | 4 | 3.6% |

### 3.1 Oracle 编译错误（典型）

LLM 生成的测试用例中引用了未定义的类型，导致 Oracle 程序编译失败：

- `error: unknown type name 'CArray'` — `array/carray.c` 中 `CArray` 类型定义在头文件中，但 Oracle 未包含
- `error: array has incomplete element type 'void'` — `void*` 参数处理不当
- `error: array has incomplete element type 'struct GraphRep'` — 图结构体定义不完整

### 3.2 空 stderr（未知错误）

约 43% 的失败函数 stderr 为空但 returncode=1。推测为 pipeline 内部异常（如 JSON 解析失败、文件 I/O 错误等），需要进一步排查。

### 3.3 返回类型不支持

`NotImplementedError: The connector currently supports scalar returns, scalar/struct pointer returns, and struct value returns.`

部分函数返回 `void**`、`struct` 数组等复杂类型，当前 pipeline 暂不支持。

## 五、关键发现与建议

### 5.1 发现

1. **LLM 能生成高质量测试用例，但环境上下文缺失**
   - 通过的函数覆盖率接近完美（94.3% 行 / 92.7% 分支）
   - 但 LLM 不知道源文件依赖的头文件和类型定义

2. **Rules 模式更稳定但覆盖较浅**
   - Rules 模式通过率 60.5%，是 Hybrid 的 5 倍以上
   - 但分支覆盖率只有 75.5%，很多边界条件未触发

3. **速度是致命瓶颈**
   - 38 秒/函数 × 294 = ~3.1 小时
   - Rules 模式仅需 ~2.5 分钟（0.6 秒/函数）

### 5.2 改进建议

1. **增强 Prompt 上下文**
   - 将目标源文件的头文件（`.h`）内容一同送入 LLM prompt
   - 让 LLM 了解所有自定义类型和结构体定义

2. **改进 Oracle 编译器**
   - 自动收集源文件所在目录的所有头文件路径
   - 对 `void*` 参数提供特殊处理（如统一转为 `char*` 或 `int*`）

3. **预过滤不支持的函数**
   - 在运行前检测返回类型，跳过不支持的函数，减少无效 LLM 调用

4. **考虑更快的 LLM 模型**
   - qwen3.5 虽然可用但响应较慢，可尝试更轻量级模型或量化版本

5. **Hybrid 模式优化策略**
   - 先用 Rules 模式跑通，再用 LLM 补充边界用例
   - 仅对 Rules 模式未覆盖的分支调用 LLM，减少调用次数
