# STRUT-Unity

This folder connects the useful pieces from `STRUT-C` and `Unity` into one runnable prototype.

- `unity/` contains Unity's core C test framework files.
- `original_strut_c/` keeps the active STRUT-C scripts for reference.
- `strut_unity/` is the connector layer: libclang extracts function context, tree-sitter checks C syntax, test cases are generated by rules and/or an OpenAI-compatible LLM endpoint, Unity tests are emitted, compiled, and run.
- The root prompt files are loaded by the pipeline:
  - `Test Cases Generation Prompts.md` drives STRUT-style structured test-suite generation.
  - `json structure.md` defines the structured case format used in prompts and `build/*_cases.json`.
  - `Test Suite Optimization Prompts Used By STRUT.md` drives the optional uncovered-branch optimization pass.
- `Prompts Used by LLM baseline method.md` is kept as baseline-comparison material and is not used by this pipeline.

Run the default hybrid demo with local Ollama:

```sh
make demo
```

The generated context, cases, Unity test file, and executable are written under `build/`.

Run only the deterministic rule generator:

```sh
make rules-demo
```

By default the LLM client uses:

```sh
STRUT_LLM_BASE_URL=http://127.0.0.1:11434/v1
STRUT_LLM_MODEL=qwen3.5:latest
```

You can also call the pipeline directly:

```sh
python3 -m strut_unity examples/classify_score.c \
  --function classify_score \
  --case-source hybrid \
  --llm-base-url http://127.0.0.1:11434/v1 \
  --llm-model qwen3.6:27b
```

`hybrid` merges deterministic boundary cases with LLM-generated cases and is the default. Use `--case-source rules` to avoid LLM calls.

LLM and hybrid modes run the STRUT optimization pass by default after the first compile/test run if extracted branch outcomes remain uncovered. Use `--no-optimize` to skip that pass.

Remote OpenAI-compatible APIs can be wired without code changes:

```sh
export STRUT_LLM_BASE_URL=https://api.openai.com/v1
export STRUT_LLM_API_KEY=...
export STRUT_LLM_MODEL=...
python3 -m strut_unity examples/classify_score.c --function classify_score --case-source llm
```

The LLM is asked to generate input arguments only. Expected outputs are still computed by compiling a small oracle program from the target C function, then the final Unity assertions are generated from those checked values.

Each run writes the function context, structured cases, generated Unity test script, compile/run output, and coverage data under `build/`. Coverage is reported with `gcov` when available. The main `line_coverage_percent` and `branch_coverage_percent` values are computed for the function under test; source-file-level values are also retained with the `source_file_` prefix. A branch-outcome estimate from extracted conditions is included as a fallback.

Supported input values now include scalar basic types such as `int` and `float`, arrays by initializing element zero, pointers by creating a valid local target or `NULL` when a null branch is visible, and struct pointers by expanding fields such as `shape->area`. Pointer expansion is intentionally bounded to two levels.
